## Rewriting `modal_qwen35_4b_t4_finetune.py`

### Problem

The original script used `build_sft_config_compat` and `build_sft_trainer_compat` wrapper functions that tried to handle TRL version differences. These wrappers passed `eos_token` and `pad_token` through `SFTConfig`, but TRL 0.22.2's `SFTConfig` defaults `eos_token` to the sentinel `'<EOS_TOKEN>'`, which isn't in the Qwen tokenizer vocabulary. This caused:

```
ValueError: The specified `eos_token` ('<EOS_TOKEN>') is not found in the vocabulary
of the given `processing_class` (TokenizersBackend).
```

Multiple fix attempts (post-construction `setattr`, `None` override) failed because `SFTConfig`'s dataclass internals don't propagate these correctly.

### Solution

Rewrote the entire script using the same proven pattern as `modal_qwen35_18b_reap_a3b_coding_opus_train.py`:
- Direct `SFTTrainer(model=model, tokenizer=tokenizer, train_dataset=..., args=SFTConfig(...))` call.
- No compat wrappers.
- Used `max_length` (not `max_seq_length`) in `SFTConfig`.
- Did not pass `eos_token`/`pad_token` to `SFTConfig` at all — let the trainer infer from the tokenizer.

Applied the same fix to `colab_qwen35_t4_finetune.py`.

## Batch Size Testing on T4

|---|---|---|---|---|

Selected **batch size 3** as a safe middle ground.

## T4 → L4 GPU Switch

- T4 does not support bf16 — Unsloth falls back to float32, resulting in ~3.4 min/step.
- Switched to L4 (24 GB, bf16 supported) — ~2.4 min/step (~30% faster).
